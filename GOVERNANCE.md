# ObzenFlow Governance

This is a lightweight governance document for an early-stage project. It exists to make decision-making, licensing intent, and maintainer responsibilities clear.

## Roles

- **Project Lead / Core Maintainer:** Kevin Webber
- **Design & UI Lead:** Emma Kellam
- **Maintainers:** Individuals with merge rights, appointed by the Project Lead based on sustained, high-quality contribution and demonstrated stewardship. 

## Contributors

Thank you to everyone who contributes to ObzenFlow.

This project does not maintain a manually-curated list of contributors in this repository (to avoid omissions and staleness). The source of truth is the git history and the GitHub contributors list.

- View contributors on GitHub: https://github.com/ObzenFlow/obzenflow/graphs/contributors
- View contributors locally: `git shortlog -sne`

If you want your name to appear correctly, configure your git `user.name` and `user.email` before contributing.


## Decision making

- Most decisions happen in issues and pull requests.
- Large or ecosystem-impacting changes should use a design proposal (FLOWIP) before implementation:
  - FLOWIPs should be submitted to the `obzenflow-improvement-proposals` repo via pull request.
  - Discussion happens in the proposal pull request until consensus is reached.
- Maintainers aim for rough consensus. If consensus can’t be reached, the Project Lead makes the final call with a written rationale.

## Contribution policy (DCO, not CLA)

- ObzenFlow uses the **Developer Certificate of Origin (DCO)**. See `DCO.md`.
- We do **not** require a Contributor License Agreement (CLA).
- Inbound = outbound: contributions are accepted under the project’s license terms (MIT OR Apache-2.0).

## Licensing philosophy and stability

ObzenFlow is dual-licensed under **MIT OR Apache-2.0**. The project’s goal is broad adoption and a healthy ecosystem.

- The project does not intend to adopt “source-available” licenses such as BSL, SSPL, or similar.
- Any license change would require an explicit public proposal and extremely high community consensus. Under the DCO, contributors retain copyright to their own work, so relicensing the repository as a whole would typically require agreement from the relevant copyright holders.
